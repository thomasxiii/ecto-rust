# EctoHost (iOS)

Swift iOS host that links the Rust ecto-engine as a static library and
renders graphs via SwiftUI.

The Swift code that actually renders the graph (RenderTreeView,
StyleMapper, SVGView, SocketManager, GraphStore) is intended to be
lifted from `~/Desktop/Projects/ecto-engine/host-ios/EctoHost/` —
nothing in that code is tied to JavaScriptCore; it consumes the
`RenderTreeNode` JSON shape that the Rust engine produces directly. The
only file that needs rewiring is `Bridge/JSCoreBridge.swift`, which is
replaced by `Bridge/EngineBridge.swift` in this repo.

## Layout

```
host-ios/
├── EctoHost/
│   └── Bridge/
│       ├── ecto_engine.h        ← generated by cbindgen
│       └── EngineBridge.swift   ← Swift wrapper around the C ABI
└── README.md                    ← you are here
```

The Xcode project itself is not committed (it'll change per-developer);
the bridging files are. To stand up the iOS app:

1. `cd ~/Desktop/Projects/ecto-rust`
2. `cargo build --release --target aarch64-apple-ios` (and/or `…-sim`
   variants if you want simulator support).
3. Open / create an Xcode project at `host-ios/EctoHost.xcodeproj`.
4. Drag `target/aarch64-apple-ios/release/libecto_engine.a` into the
   project under "Frameworks, Libraries, and Embedded Content".
5. Create a bridging header that just contains `#include "ecto_engine.h"`
   and point "Objective-C Bridging Header" at it.
6. Add `-lc++` to "Other Linker Flags" (lightningcss needs the C++ stdlib).
7. Copy `EctoHost/Views/*.swift`, `EctoHost/Models/RenderTreeNode.swift`,
   `EctoHost/Bridge/GraphStore.swift`, `EctoHost/Bridge/SocketManager.swift`
   from `~/Desktop/Projects/ecto-engine/host-ios/EctoHost/` into this
   project. The only changes needed:
   - In `GraphStore.swift`, replace
     `JSCoreBridge.shared.walkRenderTree(graphJSON: data)` with
     `try EctoEngine().loadGraph(data); let tree = try engine.walkRenderTree(rootId: entryId)`.
   - Drop the `core.bundle.js` resource and the `JSCoreBridge.swift` file.

## Why FFI instead of JavaScriptCore

The original iOS host bundles `@ecto/core` as JS and runs it through
JavaScriptCore. That works but means:
- The engine has two implementations (TS and Swift's render side)
- JS↔Swift marshalling is per-call JSON, same overhead as FFI
- Updates require rebuilding the JS bundle and shipping it as a resource

The Rust engine compiled to a native arm64 static lib:
- Same source as the WASM build that runs in the browser
- Strictly faster than JSCore for graph traversal
- Smaller binary footprint (no JSCore runtime)
- Same JSON contract, so existing Swift rendering code is unchanged

## Cross-targets

```
rustup target add aarch64-apple-ios          # iPhone / iPad (device)
rustup target add aarch64-apple-ios-sim      # iOS Simulator on Apple Silicon
rustup target add x86_64-apple-ios           # iOS Simulator on Intel Macs
```

A `lipo` step at the end can fatten device + sim into one
universal `.a` if you want a single drop-in. Example:

```bash
lipo -create \
  target/aarch64-apple-ios/release/libecto_engine.a \
  target/aarch64-apple-ios-sim/release/libecto_engine.a \
  -output libecto_engine_universal.a
```

## Regenerating the header

```bash
cbindgen --crate ecto-engine --lang c \
  --output host-ios/EctoHost/Bridge/ecto_engine.h
```
